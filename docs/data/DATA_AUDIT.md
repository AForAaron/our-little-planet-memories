# Datas 数据体检与网页导入建议

检查日期：2026-07-06

## 结论

这批数据适合做网站的“原始导入源”，但不适合原样放进网页，也不应把 67,553 条聊天全部写入 `entries`。

- `Datas/Photos` 适合清洗后作为旅行/时间轴相册。
- 微信 JSON 结构完整，消息与已导出媒体的关联可靠，适合写离线 ETL 脚本。
- 网站首期应只导入“精选记忆”，而不是做完整微信聊天镜像。
- 原始 JSON、ZIP、WAV、EXE 不应进入 Next.js 仓库、Vercel 构建产物或公开静态目录。

## 数据概况

整个 `Datas` 约 9.1GB。

### 独立相册

- 300 张 JPG
- 4 个 MP4
- 约 1.6GB
- 文件名时间范围：2026-04-04 至 2026-05-14
- 单张图片约 1.7MB–22.4MB，平均约 4.74MB

### 微信导出

- 1 个会话
- 67,553 条消息
- 时间范围：2026-03-08 13:05:56 至 2026-07-06 12:30:38
- `messages.json` 约 90MB
- 媒体目录约 2.8GB
- 828 张聊天图片
- 17 个视频和 17 张视频封面
- 3,102 条语音，总时长约 14.53 小时
- 4 个已导出文件
- 3,596 个未导出媒体，其中 3,586 个是表情、10 个是文件

两份 ZIP 的导出统计完全相同，后一份还存在完整解压目录。它们很可能是同一批数据的重复导出，当前合计占用了大量重复空间；在确认内容哈希之前不要直接删除。

## 图片检查

抽样观察到两类明显不同的内容：

1. `Datas/Photos`：高分辨率旅行风景、双人合照、连续拍摄照片，适合成为正式时间轴和旅行相册。
2. `media/images`：聊天中发送的照片、截图、网络图片、梗图和学习资料混合，不能全部当成“共同回忆照片”。

对 1,128 张 JPG/PNG 做了解码检查：

- 1,126 张 JPEG、2 张 PNG
- 607 张竖图、229 张横图、292 张方图
- 319 张最长边超过 3,000px
- 没有发现损坏或无法解码的图片
- 最小尺寸只有 67×56，最大高度达到 16,330px，说明其中包含缩略图和长截图
- 发现 2 组完全相同的聊天图片；相册中还有多组同秒或 `_01` 连拍，需做近似去重/人工挑选

### 图片不应直接原样上网页

独立相册平均 4.74MB，直接用于时间轴会明显拖慢手机加载。建议：

- R2 可选保存原图；
- 额外生成网页展示版：最长边 2,048 或 2,560px；
- 输出 WebP，质量约 80–85；
- 再生成 480px 或 640px 缩略图用于列表；
- 数据库分别记录原图 key、展示图 key、缩略图 key，或者首期只上传展示版；
- 保留宽、高、MIME、文件字节数和原始文件哈希；
- 不根据文件扩展名判断 MIME，导入时读取真实文件头。

## 聊天 JSON 是否适合

### 适合的部分

`messages.json` 是合法的结构化 JSON，每条消息具备：

- 稳定且唯一的字符串 `id`
- `createTime` 秒级时间戳
- `sortSeq` 毫秒排序值
- `renderType`
- `isSent`
- 发送者 username / display name
- 文本内容
- 引用消息字段
- `offlineMedia[].path` 本地媒体引用

3,968 个唯一媒体引用全部能在导出目录找到；额外两个未引用文件只是双方头像。因此“消息 → 本地媒体”的关联质量很好。

### 必须处理的问题

1. **不能由浏览器直接加载 90MB JSON**
   - 必须在本地或受保护的后台离线解析；
   - 网页只请求分页后的数据库结果。

2. **不要用 JavaScript `number` 保存 `serverId`**
   - 67,497 条消息的 `serverId` 超过 `Number.MAX_SAFE_INTEGER`；
   - `JSON.parse` 后可能已失去精度；
   - 去重键应直接使用完整字符串 `id`；
   - 写入 `entries.source_ref` 时也用 `id`，不要用数值 `serverId`。

3. **时间文本没有时区**
   - `createTimeText` 是本地格式；
   - 导入时以 `createTime` 为准，并明确按 `Asia/Shanghai` 展示；
   - `happened_at` 写入 PostgreSQL `timestamptz`。

4. **语音长度单位标注错误**
   - `voiceLength` 实际看起来是毫秒；
   - 示例 `27401` 应理解为约 27.4 秒，而不是 27,401 秒；
   - UI 不应直接使用导出 JSON 中的 `content` 文案。

5. **语音不适合原样用于网页**
   - 3,095 个 WAV 占约 2.3GB，另有 7 个 SILK；
   - WAV 体积过大，SILK 不是浏览器通用格式；
   - 如后续支持语音，应统一转为 AAC/M4A 或 Opus/WebM，并保留时长；
   - 首期时间轴不建议导入全部语音。

6. **表情基本缺失**
   - 3,586 个表情都没有离线文件；
   - 远程 `emojiUrl` 不应被当成长期可靠资源；
   - 首期可显示为 `[表情]` 或忽略，不要阻塞导入。

7. **文件存在安全风险**
   - 已导出文件里包含两个 Windows `.exe`；
   - 禁止上传、托管或提供下载；
   - 文件类型首期采用白名单，最好只允许图片、视频和明确需要的文档格式。

8. **导出时没有启用隐私模式**
   - JSON 包含真实微信 ID、昵称、头像路径、消息内容及引用内容；
   - 不得提交到 Git、放进 `public/`、发送给客户端或写入日志；
   - `.gitignore` 当前没有忽略 `Datas`，后续正式建立 Git 仓库前必须增加 `/Datas/`。

## 不建议直接映射到 `entries`

`entries` 是“精选共同记忆”，一条记录应当是一个有意义的事件。67,553 条消息如果全部写入：

- 时间轴会被短句、表情、引用回复淹没；
- 一次真实事件会被拆成几十到几百条碎片；
- 图片、视频和语音很难形成可读叙事；
- `entries.body` 会失去“人工整理后的回忆”定位。

推荐采用两步流程：

### 第一步：本地生成清洗候选集

按 30–90 分钟无消息作为会话分段边界，再结合：

- 同一天/同一事件连续聊天
- 图片或视频
- 关键词：第一次、见面、旅行、生日、纪念日、想去、好吃等
- 较长文本
- 双方都参与的对话段

输出一个轻量候选文件，只包含：

```json
{
  "candidate_id": "wechat-segment-...",
  "start_at": "2026-04-07T...",
  "end_at": "2026-04-07T...",
  "title_suggestion": "",
  "summary": "",
  "message_ids": ["完整字符串 id"],
  "media_paths": ["media/images/..."],
  "selected": false
}
```

### 第二步：人工审核后写入网站

每个被选中的事件转成一条 `entries`：

- `source = "wechat_import"`
- `source_ref = candidate_id` 或主消息完整字符串 id
- `happened_at = 事件发生时间`
- `title/body = 人工确认后的标题和叙述`
- 精选图片上传 R2 后写入 `media`

这样可以保留微信来源和防重复能力，又不会把网站做成聊天日志仓库。

## 如果未来要展示完整聊天

当前 brief 的六张表不适合保存完整聊天明细。如果未来目标变成“按日期浏览、搜索全部聊天”，应另行设计：

- `chat_imports`
- `chat_conversations`
- `chat_messages`
- `chat_message_media`

这属于独立功能，不应和首期时间轴迁移同时实施。

## 推荐清洗顺序

1. 先把 `/Datas/` 加入 `.gitignore`。
2. 保留一份原始 ZIP 作为不可变备份。
3. 对两份 ZIP 做 SHA-256 和内容清单对比，确认后再决定是否删除重复副本。
4. 建立只读导入脚本，永远不修改原始数据。
5. 图片按 SHA-256 精确去重，再按感知哈希发现连拍/近似图。
6. 对独立相册按文件名时间分组，人工选择每个事件的封面和精选图。
7. 聊天按会话段生成候选事件，不导入全部消息。
8. 先用 10–20 个事件做小批量试导入。
9. 验证时间、人物、图片方向、R2 签名和重复导入。
10. 小批量稳定后再扩展。

## 对网站数据结构的直接影响

新版 brief 里的 `source` / `source_ref` 足以支持“精选微信事件导入”。但图片优化后可能需要区分原图和展示图；在保持现有模型不变的前提下，首期最简单的方式是只上传优化后的展示图，把原图继续保留在离线备份中。

这比立即扩展 `media` 表更稳妥，也更符合当前“先跑通时间轴纵切片”的范围。
